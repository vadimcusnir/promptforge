




În contextul PROMPTFORGE™, „7D” înseamnă motorul de parametrizare pe 7 dimensiuni. Este schema canonică după care se construiește orice rulare (prompt, modul, artefact). Cele 7 dimensiuni sunt:

Domain – industria / verticala (din nomenclatorul CORE-25: SaaS, FinTech, E-Commerce, Education, Healthcare etc.).

Scale – nivelul de scară: personal_brand, solo, startup, SMB, corporate, enterprise.

Urgency – gradul de urgență: low, planned, sprint, pilot, crisis.

Complexity – nivelul de complexitate: foundational, standard, advanced, expert.

Resources – resurse disponibile: minimal, solo, lean_team, agency_stack, full_stack_org, enterprise_budget.

Application – aplicația concretă: training, audit, implementation, strategy, crisis_response, experimentation, documentation.

Output – formatul livrabilului: txt, md, checklist, spec, playbook, json, yaml, diagram, bundle, pdf.

🔑 Rolul 7D:
– Standardizează inputul pentru toate modulele (schema JSON).
– Garantează consistență și validare (enum-only, guardrails în ruleset.yml).
– Decide gatingul (ce plan poate exporta anumite formate).
– Permite măsurarea KPI (time-to-artifact, scor evaluare, etc.).

Pe scurt: 7D = motorul semantic și operațional al PROMPTFORGE™. Fiecare prompt/modul rulează numai dacă are un set valid de 7D (Definition of Ready), și este acceptat ca „Done” doar dacă artefactul rezultat respectă și scorurile + exportul (Definition of Done).






🧩 1. Domain (Industrie / Verticală)

📚 Nomenclator CORE-25 (definit deja în schema ta):

SaaS → sub-industrii: B2B SaaS, Productivity, Collaboration, DevTools

FinTech → Banking, Payments, Lending, Insurance, RegTech

E-Commerce → Retail, Marketplaces, D2C, Dropshipping

Consulting → Management, Strategy, Marketing, IT Consulting

Education → E-learning, K-12, Higher Ed, Corporate Training

Healthcare → Hospitals, Telemedicine, MedTech, Pharma

… (până la Agriculture, Gaming, Beauty, Fashion etc.)

🔑 Sub-parametri Domain:

sub_industry (array enum) – ex: “Digital Banking”, “MOOCs”, “InsurTech”

jargon[] – termeni tehnici specifici (ARR, KYC, AOV…)

kpis[] – indicatori de performanță canonici (MRR growth, fraud_rate, completion_rate)

compliance_notes – reguli legale sau restricții

style_bias – ton recomandat (audit, narativ, formal…)

default_output_format – spec / playbook / plan

🧩 2. Scale (Nivel de scară)

Valori de bază: personal_brand, solo, startup, SMB, boutique_agency, corporate, enterprise.

🔑 Sub-parametri Scale:

team_size → numeric range (1, 2–5, 10–50, 50+)

budget_band → low / medium / high

market_scope → local / regional / global

decision_structure → founder-driven / board-driven / committee

🧩 3. Urgency (Gradul de urgență)

Valori de bază: low, planned, sprint, pilot, crisis.

🔑 Sub-parametri Urgency:

timeline → zile/săpt./luni (ex: 7d, 30d, 90d)

pressure_type → market / regulatory / competitive / internal

risk_tolerance → low / medium / high

decision_speed → instant / fast-track / deliberative

🧩 4. Complexity (Nivel de complexitate)

Valori de bază: foundational, standard, advanced, expert.

🔑 Sub-parametri Complexity:

steps_required → 3, 5, 10+

stakeholders_count → 1, 2–3, 5+

tools_involved → single / multi-stack

knowledge_depth → novice / intermediate / specialized / research-grade

failure_cost → low / medium / high

🧩 5. Resources (Resurse disponibile)

Valori de bază: minimal, solo, lean_team, agency_stack, full_stack_org, enterprise_budget.

🔑 Sub-parametri Resources:

human → solo / 2–3 ppl / department / cross-org

financial → bootstrap / seed / VC-funded / enterprise budget

tech_stack → lightweight / integrated SaaS / custom infra

time_allocation → <5h/w, 10–20h/w, full-time

outsourcing → none / partial / full

🧩 6. Application (Aplicație concretă)

Valori de bază: training, audit, implementation, strategy, crisis_response, experimentation, documentation.

🔑 Sub-parametri Application:

artifact_type → curriculum, report, SOP, plan, playbook

end_user → student, employee, manager, regulator, client

delivery_mode → async, sync, blended, automated

measurement_metric → ROI, retention, compliance, throughput

maturity_level → POC, MVP, Pilot, Production, Scale

🧩 7. Output (Formatul livrabilului)

Valori de bază: txt, md, checklist, spec, playbook, json, yaml, diagram, bundle, pdf.

🔑 Sub-parametri Output:

file_format → txt, md, json, pdf, zip

structure → narrative, table, diagram, SOP steps

visuals → text-only, with charts, with branding

validation → checksum required, manifest required, score≥80 required

retention → local only, cloud 30d, cloud 90d, archive permanent

⚙️ Cum folosești sub-parametrii?

În Supabase schema → fiecare parametru 7D are sub-coloane în JSONB (config).

În UI (React/Next) → filtre avansate, form controls (sliders, selects, tags).

În Evaluator AI → scoruri calibrate diferit per sub-parametri (ex: urgency=crisis → toleranță ambiguitate=0).

îți detaliez acum 📍 1.2 Sub-parametri Domain. Fiecare domeniu (din CORE-25) nu este doar o etichetă, ci o structură parametrică completă cu sub-câmpuri obligatorii. Acestea devin meta-date canonice pentru fiecare domain_config din baza ta de date (Supabase/Postgres) și pentru validarea JSON Schema din module.spec.json.

🔑 Sub-parametri Domain
1. sub_industry (array enum)

Rol: definește granularitatea domeniului.

Tip: string[] (enumuri predefinite, max 10 per domain).

Exemple:

SaaS → [“B2B SaaS”, “Productivity Tools”, “Collaboration Platforms”, “DevTools”]

FinTech → [“Digital Banking”, “Payments & Wallets”, “InsurTech”, “RegTech”]

Education → [“E-learning”, “K-12”, “Higher Ed”, “Corporate Training”]

👉 Beneficiu: permite filtrare și personalizare în UI (dropdown, multi-select), plus analytics mai fin pe segmente.

2. jargon[] (termeni tehnici specifici)

Rol: vocabularul propriu industriei, injectat în prompts ca context.

Tip: string[] (min 5 – max 50 termeni).

Exemple:

SaaS → ARR, MRR, churn, CAC, NRR, multi-tenant, API-first

FinTech → KYC, AML, PSD2, open banking, risk scoring

E-Commerce → CR, AOV, ROAS, cart abandonment, SKU

👉 Beneficiu: engine-ul poate valida dacă outputul folosește jargonul corect și poate ajusta stilul.

3. kpis[] (indicatori canonici de performanță)

Rol: setul de KPI-uri care trebuie să apară în livrabil (scoruri, planuri).

Tip: string[] (min 3 – max 20 indicatori).

Exemple:

SaaS → [“MRR growth”, “churn rate”, “time-to-value”]

FinTech → [“fraud_rate”, “KYC_completion_time”, “approval_rate”]

Education → [“completion_rate”, “engagement_rate”, “retention_rate”]

👉 Beneficiu: evaluatorul AI poate nota “business_fit” verificând apariția acestor KPI în output.

4. compliance_notes

Rol: regulile și restricțiile legale pentru domeniu.

Tip: text (până la 2000 caractere).

Exemple:

SaaS → GDPR, SOC2, ISO27001, Data Residency (EU/US).

FinTech → SEC/FCA sensitive, Basel III, PCI DSS, Anti-Money Laundering.

Healthcare → HIPAA, GDPR, FDA/EMA approvals.

👉 Beneficiu: se injectează ca guardrail în prompts → engine știe ce să evite (ex: fără “profit garantat”, fără “diagnostic medical”).

5. style_bias

Rol: definește tonul și vocea recomandată pentru livrabile.

Tip: enum string (audit | narativ | formal | empatic | persuasiv | experimental).

Exemple:

SaaS → analitic, orientat pe metrici.

Consulting → audit-like, autoritar.

Education → narativ, gamified, empatic.

FinTech → formal, layered, risc/reglementare.

👉 Beneficiu: influențează micro-copy în output → modulul știe dacă să fie mai “storytelling” sau mai “report-like”.

6. default_output_format

Rol: setul de livrabile standard pentru industrie.

Tip: enum string (txt | md | checklist | spec | playbook | json | pdf | bundle).

Exemple:

SaaS → spec (documentație tehnică).

Marketing → playbook (funneluri, campanii).

Healthcare → spec (protocoale clinice).

Education → playbook (curriculum, rubrici).

👉 Beneficiu: engine-ul propune implicit formatul cel mai util pentru acel domeniu, dar permite override.

📑 Exemplu JSON pentru un domain_config (FinTech)
{
  "id": "fintech",
  "industry": "FinTech",
  "sub_industry": ["Digital Banking", "Payments", "InsurTech", "RegTech"],
  "jargon": ["KYC", "AML", "PSD2", "open banking", "risk scoring"],
  "kpis": ["fraud_rate", "KYC_completion_time", "approval_rate"],
  "compliance_notes": "SEC/FCA sensitive; Basel III; PCI DSS; GDPR; AML directives.",
  "style_bias": "formal",
  "default_output_format": "spec"
}


🔧 Integrare practică:

În Supabase → coloane sub_industry[], jargon[], kpis[], compliance_notes text, style_bias text, default_output_format text.

În UI → filtre de tip select/multi-select (sub_industry), checklist (KPI), textarea (compliance).

În Evaluator → se verifică dacă jargonul și KPI apar în output + se aplică tonul cerut.








🧩 2. Scale (Nivel de scară)
🎯 Valori canonice (enum de bază)

personal_brand → 1 individ, branding propriu.

solo → freelancer, one-man-show cu câteva tool-uri.

startup → echipă 2–15 oameni, finanțare seed/pre-seed.

SMB → 15–100 oameni, business stabilizat, creștere sănătoasă.

boutique_agency → 5–30 oameni, clienți multipli, servicii creative/consulting.

corporate → sute/mii angajați, proceduri, bugete mari.

enterprise → global, multi-divizii, infrastructură complexă.

🔑 Sub-parametri Scale
1. team_size

Rol: definește resursa umană disponibilă.

Tip: interval numeric sau enum scalat.

Valori recomandate:

1 → personal brand / solo

2–5 → startup foarte mic

6–15 → startup / boutique agency

16–50 → SMB

51–250 → corporate mid-size

250+ → enterprise

👉 Impact: prompts și SOP-urile trebuie să includă roluri și responsabilități adaptate numărului de oameni.

2. budget_band

Rol: definește resursele financiare disponibile.

Tip: enum {low, medium, high}.

Mapping sugestiv:

Low: bootstrapped, <10k €/lună buget operațional.

Medium: 10k–100k €/lună (SMB, startup cu funding).

High: 100k+/lună (corporate, enterprise).

👉 Impact: decide ce tool-uri/stack-uri sunt recomandate (no-code vs enterprise SaaS vs custom infra).

3. market_scope

Rol: aria de piață vizată.

Tip: enum {local, regional, global}.

Definiții:

Local: 1 țară / comunitate restrânsă.

Regional: UE, APAC, LatAm etc.

Global: prezență internațională, multi-limbă, multi-tax.

👉 Impact: prompts pentru marketing/branding includ localizare, i18n, reglementări fiscale.

4. decision_structure

Rol: modul în care se iau deciziile în organizație.

Tip: enum {founder-driven, board-driven, committee}.

Definiții:

Founder-driven: 1–2 persoane decid rapid (solo/startup).

Board-driven: investitori și board decid (SMB / startup cu funding).

Committee: departamente, procese lente (corporate, enterprise).

👉 Impact: prompts trebuie să includă „layers of approval” și moduri de prezentare (ex: business case pentru board vs one-pager pentru fondator).

📑 Exemplu JSON — Scale pentru un startup
{
  "scale": "startup",
  "team_size": "6-15",
  "budget_band": "medium",
  "market_scope": "regional",
  "decision_structure": "founder-driven"
}

🧭 Integrare practică

Supabase/DB: coloane team_size text, budget_band text, market_scope text, decision_structure text.

UI: selectori dropdown / sliders pentru team size și buget.

Evaluator AI: ajustează scorul „business_fit” în funcție de scale (ex: corporate → output mai formal; personal_brand → limbaj mai autentic).








🧩 3. Urgency — enum de bază

low – fără presiune imediată; explorare/învățare.

planned – proiect programat; termene clare dar flexibile.

sprint – fereastră scurtă (1–3 săpt.); livrabile iterativ.

pilot – probă controlată cu criterii de reușită; impact moderat.

crisis – incident/acuțiune; zero ambiguitate, decizie rapidă.

🔑 Sub-parametri
1) timeline

Rol: încadrează livrabilele în timp (zile/săpt./luni).

Tip & exemple: string tipat ca „7d”, „30d”, „90d”.

Heuristici:

low → 60–90d

planned → 30–60d

sprint → 7–21d

pilot → 14–45d (include milestones)

crisis → 24–72h

Efecte practice: dimensionează granularitatea prompturilor (SOP scurt vs. plan complet) și limitează numărul de pași/artefacte per run.

2) pressure_type

Rol: natura presiunii ce dictează stilul și guardrails.

Tip: enum {market, regulatory, competitive, internal}

Guidelines:

market – livrabile orientate pe conversie/time-to-value.

regulatory – includere explicită de conformitate, citări și checklist; export .json/.pdf preferat.

competitive – comparativ/benchmark + roadmap scurt.

internal – focus pe aliniere și adoptare (playbooks).

3) risk_tolerance

Rol: cât risc acceptă organizația în decizii & output.

Tip: enum {low, medium, high}

Aplicare:

low → guardrails stricte, praguri evaluator mai sus, mai puține „salturi” de ipoteze.

medium → standardele implicite 7D.

high → permite experimente (ex: 2 variante de prompt în paralel).

Legat de DoD: pentru risk_tolerance=low, ridică minimele de scor și blochează export dacă lipsesc secțiuni „evidence/source”.

4) decision_speed

Rol: cât de repede trebuie luată decizia.

Tip: enum {instant, fast-track, deliberative}

Pattern-uri:

instant – 1 pager + checklist; fără ambiguitate; fallback-uri predefinite.

fast-track – brief + 2 opțiuni cu criterii clare.

deliberative – raport comparativ, riscuri, cost/benefit; anexe.

Mape de comportament (implicite în engine)
Urgency	Timeline	Pressure default	Risk tol.	Decision speed	Stil & Output implicit
low	60–90d	internal/market	medium	deliberative	narativ + plan de studiu
planned	30–60d	internal	medium	fast-track	spec/playbook echilibrat
sprint	7–21d	competitive	medium	fast-track	SOP scurt, milestones
pilot	14–45d	market/regulatory	medium	fast-track	plan pilot + criterii
crisis	24–72h	regulatory/competitive	low	instant	checklist & roluri clare

Notă: Aceste preseturi se leagă de gating (formate permise pe plan) și de tipurile de artefacte (ex: regulatory → .pdf/.json) conform entitlements/export policy din sistem.

Praguri dinamice de calitate (Test Engine)

Integrează Urgency în rubrică (Claritate/Execuție/Ambiguitate/Business-Fit), ajustând pragurile DoD:

crisis: Claritate ≥ 90, Execuție ≥ 90, Ambiguitate ≤ 10, Business-Fit ≥ 80; export blocat dacă lipsește secțiunea „Escalation Path”.

sprint/pilot: Claritate ≥ 85, Execuție ≥ 85, Ambiguitate ≤ 15, Business-Fit ≥ 78.

planned/low: pragurile standard (ex.: ≥80 / ≤20) rămân aplicabile.

Micro-reguli de generare (stil & structură)

crisis → „directive, binar, fără floricele”; „Next step in 60 minutes”, owner + SLO; Output: checklist + timeline.

sprint → „milestone-driven”; Output: SOP pe 7–14 zile + criterii de acceptare.

pilot → include „Success Criteria”, ipoteze & metrici; Output: plan pilot + șablon raport.

planned/low → documentație completă, alternative, riscuri & mitigații.

Exemple
A) Urgency = crisis (regulatory, low risk, instant)

  "urgency": "crisis",
  "timeline": "72h",
  "pressure_type": "regulatory",
  "risk_tolerance": "low",
  "decision_speed": "instant"
}


Efecte:
– Forțează format „checklist/spec” + .pdf/.json la export (dacă planul permite).
– Praguri DoD ridicate + blocare export fără „Escalation Path” și „Contact Matrix”.

B) Urgency = sprint (competitive, medium risk, fast-track)
{
  "urgency": "sprint",
  "timeline": "14d",
  "pressure_type": "competitive",
  "risk_tolerance": "medium",
  "decision_speed": "fast-track"
}


Efecte:
– Output: SOP cu 3 milestones, 2 opțiuni tactice, criterii clare; evaluarea cere execuție ≥85.

{
  "type": "object",
  "properties": {
    "urgency": { "type": "string", "enum": ["low","planned","sprint","pilot","crisis"] },
    "timeline": { "type": "string", "pattern": "^(\\d+)(d|w|m)$" },

Integrare în pipeline

DoD/Export: la crisis & regulatory, cere .pdf/.json + manifest/checksum pe bundle; altfel 422 + hint de corecție.








🧩 4. Complexity — enum de bază

standard — flux obișnuit cu câteva dependențe.

advanced — mai multe părți mobile, integrații, validări.

expert — arhitectură, cercetare sau conformitate strictă; risc mare dacă greșești.

Efecte:

standard → medium

– exportul .pdf/.json devine obligatoriu în domenii reglementate (dacă entitlements permit).

🧩 5. Resources — enum de bază

minimal — resurse abia suficiente (timp/bani/unelte foarte limitate).

solo — o singură persoană executantă cu câteva tool-uri.

lean_team — 2–5 oameni, eficient, fără birocrație.

agency_stack — echipă mică-medie cu tool-uri profesionale și procese.

full_stack_org — departamente separate (produs, marketing, legal, data).

enterprise_budget — bugete mari, multi-vendor, cerințe enterprise.

🔑 Sub-parametri (definiții + efecte în engine)
1) human

Tip: enum: solo | 2–3 ppl | department | cross-org
Rol: capacitatea umană disponibilă pentru execuție.
Mapare tipică după Resources:

minimal/solo → solo

lean_team → 2–3 ppl

agency_stack → 2–3 ppl / department

full_stack_org → department

enterprise_budget → cross-org
Efecte:
– generatorul include/omite roluri și responsabilități; la department/cross-org impune o mini-secțiune RACI (Owner/Consulted/Informed) în livrabil (condiție DoD în proiectele complexe).

2) financial

Tip: enum: bootstrap | seed | VC-funded | enterprise budget
Rol: constrângeri/opțiuni financiare.
Heuristici de selecție a soluțiilor:

bootstrap/seed → preferă no-code / SaaS accesibile, evită tooling scump;

VC-funded → acceptă un mix SaaS + integrații plătite;

enterprise budget → permite stack multi-vendor, SSO, audit, exporturi enterprise.
Efecte:
– influențează recomandările de tooling și formatele de export (PDF/JSON/ZIP dacă entitlements permit).

3) tech_stack

Tip: enum: lightweight | integrated SaaS | custom infra
Rol: maturitatea și complexitatea infrastructurii.
Ghid:

lightweight (Notion/GSheets/Zapier) – setup rapid, cost mic;

integrated SaaS (HubSpot/SFDC/+ BI) – rapoarte, API, procese;

custom infra (microservicii, queue, data lake) – control total + cost/efort mare.
Efecte:
– la custom infra, generatorul cere tabele de compatibilitate și fallback-uri pe integrații (DoR/DoD);
– la lightweight, forțează SOP curt + checklists în loc de arhitecturi grele.

4) time_allocation

Tip: enum: <5h/w | 10–20h/w | full-time
Rol: timpul real disponibil săptămânal.
Efecte:
– <5h/w → livrabile strict esențiale (3 pași, 1 artefact);
– 10–20h/w → SOP/plan în 5 pași cu milestones;
– full-time → plan complet + experimente A/B + raport.

Engine-ul restrânge numărul de pași/artefacte în funcție de timp, pentru a respecta DoD (nu livrăm scope imposibil).

5) outsourcing

Tip: enum: none | partial | full
Rol: externalizare execuție.
Efecte:
– la partial/full, prompturile includ „handoff package” (brief, criterii, fișiere sursă, check de calitate);
– Test Engine penalizează Ambiguity dacă lipsesc clarificări pentru handoff (praguri DoD).

agency_stack	2–3 / dept	VC-funded	integrated SaaS	full-time	partial
full_stack_org	department	VC-funded	custom infra	full-time	partial
enterprise_budget	cross-org	enterprise budget	custom infra	full-time	full
Reguli dinamice (legate de Resources)

Export policy:
– enterprise_budget + custom infra → recomandă .pdf/.json și, dacă planul permite, bundle .zip cu manifest+checksum (audit, QA).

Praguri calitate (exemplu):
– minimal/solo → Clarity/Execution minime standard, focus pe 1 artefact;

🧩 6. Application (Aplicație concretă) și îl legăm de reguli executabile în generator, DoR/DoD și export. Reamintesc: 7D este schema canonică a rulărilor, validată la intrare (DoR) și la ieșire (DoD) în pipeline-ul tău; formatele de export și gating-ul pe planuri sunt deja standardizate în app/API/UI.

🧩 6. Application — enum de bază

training — instruire, învățare, curriculum.

audit — verificare/diagnostic cu dovezi.

implementation — execuție operațională (SOP, runbook).

strategy — direcție & arhitectură decizională.

crisis_response — acțiune rapidă, zero ambiguitate.

experimentation — teste, A/B, ipoteze.

documentation — specificații, ghiduri, referințe.

🔑 Sub-parametri (definiții + efecte în engine)
1) artifact_type

Tip: enum: curriculum | report | SOP | plan | playbook
Rol: formatul principal al livrabilului (în afara extensiilor de export).
Reguli:

training → curriculum / playbook

audit → report (secțiuni „Findings / Evidence / Risk / Remediation”)

implementation → SOP (pași, owners, SLO, fallback)

strategy → plan (opțiuni, criterii, roadmap)

crisis_response → SOP + „Escalation Path”

experimentation → report (ipoteze, design, rezultate, decizie)

documentation → spec/guide (în practică „report/plan” cu structură fixă)

Efect în DoD: impune secțiuni obligatorii diferite (ex.: audit cere „Evidence” și „Severity Matrix”; crisis cere „Escalation Path”).

2) end_user

Tip: enum: student | employee | manager | regulator | client
Rol: pentru cine e scris livrabilul → ton, granularitate, dovezi.
Guidelines:

student/employee → exemple & exerciții / pași executabili

manager/client → sumar executiv, KPI, opțiuni + recomandare

regulator → citări, conformitate, anexe, trasabilitate

Efect în generator: ajustează micro-copy, nivelul de detaliu și secțiunile „Summary vs. Deep Dive”.

3) delivery_mode

Tip: enum: async | sync | blended | automated
Rol: modul de livrare/execuție.
Aplicare:

async → materiale consumabile independent (docs, video, checklists)

sync → agende, ritm, timpi, facilitare (meeting playbooks)

blended → mix; cere timeline și repere de verificare

automated → pași + integrare (webhooks/jobs) și criterii de retry

Efect în DoR/DoD: pentru automated cere secțiune „Triggers, Inputs, Outputs, Error Handling” (altfel respins la DoR).

4) measurement_metric

Tip: enum: ROI | retention | compliance | throughput
Rol: ce optimizăm și cum validăm reușita.
Minime pe tip:

ROI → cost vs. beneficiu, payback, ipoteze

retention → cohorte, D1/D7/D30, NRR

compliance → checklist normativ, evidențe, deviații

throughput → p95 timp/volum, blocanți, capacitate

Efect în evaluator: Business-Fit verifică prezența KPI-urilor și a formulelor/criteriilor de măsurare în artefact.

5) maturity_level

Tip: enum: POC | MVP | Pilot | Production | Scale
Rol: stadiul inițiativei → câtă robustețe și câtă documentație sunt necesare.
Ancore:

POC → rapid, ipoteze, risc acceptat, fără „hardening”

MVP → minim funcțional + criterii de acceptare

Pilot → grup țintă + „Success Criteria” și raport pilot

Production → SLO, monitoring, rollback, runbooks

Scale → multi-regiuni/echipe, training, audit periodic

Efect în export: Production/Scale recomandă .pdf/.json și bundle cu manifest+checksum (dacă planul permite) pentru trasabilitate.

🧩 7. Output — enum de bază

Valori: txt, md, checklist, spec, playbook, json, yaml, diagram, bundle, pdf.

Formatul final livrabil (și mixul de fișiere exportate) este controlat de plan/entitlements și de policy-ul de export din app/API.

🔑 Sub-parametri Output (definiții + efecte în engine)
1) file_format

Tip: enum: txt | md | json | pdf | zip
Rol: extensia/embalarea exportului.
Gating (implicit în sistem):

Free → txt (doar)

Creator → txt, md

Pro → txt, md, pdf, json

Enterprise → + zip (bundle complet)
Efect: generatorul activează/ascunde opțiuni, iar API blochează exporturi nepermise pe plan (422).

2) structure

Tip: enum: narrative | table | diagram | SOP steps
Rol: forma conținutului în fișier (independent de extensie).
Heuristici:

narrative → rapoarte/spec (secțiuni: Context→Findings→Actions)

table → matrice KPI, mapări tool↔task, checklists

diagram → diagrame (mermaid/plantUML) în md/pdf

SOP steps → pași numerotați, owner/SLO, fallback
Efect DoD: pentru SOP steps, evaluatorul cere pași numerotați + owner + criterii; altfel export blocat.

3) visuals

Tip: enum: text-only | with charts | with branding
Rol: nivel vizual în export.
Aplicare:

with charts → anexe cu grafice (md→pdf), legate de measurement_metric

with branding → header/footer, logo, paletă (plan ≥ Pro recomandat)
Efect export: pdf compilează stilurile; md/txt rămân plain. (UI are componente dedicate pentru preview/export.)

4) validation

Tip: set flags: checksum required, manifest required, score≥80 required
Rol: condiții obligatorii înainte de livrare.
Politici standard:

manifest & checksum: pentru bundle și exporturi enterprise; se calculează SHA256 per fișier și pachet.

score≥80: blocare export (PDF/JSON/ZIP) sub prag (gate DoD).
Efect API: dacă flag-ul este cerut, handler-ul refuză exportul cu 422 și motivul (ex: scor insuficient; manifest/ checksum lipsă).

5) retention

Tip: enum: local only | cloud 30d | cloud 90d | archive permanent
Rol: păstrarea artefactelor.
Mapping implicit pe plan:

Free → local only (istoric local, fără cloud)

Creator → cloud 30d

Pro → cloud 90d

Enterprise → archive permanent (politici org/tenant)
Efect: salvează în runs/exports cu TTL/archivare; Enterprise folosește bundle cu manifest+checksum pentru trasabilitate.

Conținut: prompt.txt, prompt.md, prompt.json, prompt.pdf, telemetry.json, checksum.txt, manifest.json.

Manifest: bundle_id, run_id, module_id, version, formats, lista de fișiere cu SHA256, timestamp; opțional semnătură ed25519.

Naming & storage: /exports/{org_id}/{module_id}/{run_hash}/… + bundle-{module_id}-{run_hash}.zip.

Praguri dinamice DoD (legate de Output)

pdf/json/zip → score≥80 obligatoriu (altfel 422).

zip (bundle) → manifest & checksum obligatorii; lipsă ⇒ 422; Pro/Ent only.

SOP steps → pași numerotați + owner + SLO + fallback; lipsă ⇒ 422.

b) Constrângeri minime pe config (enum-only, pattern timeline)

Postgres nu „știe” JSON Schema nativ, dar poți face garduri simple cu jsonb_path_exists:

--  Urgency.timeline = '7d|30d|90d|...'
alter table parameter_sets
  add constraint ck_urgency_timeline
  check (jsonb_path_exists(config, '$.urgency.timeline ? (@ like_regex "^[0-9]+(d|w|m)$")'));

--  Decision speed ∈ {instant, fast-track, deliberative}
alter table parameter_sets
  add constraint ck_decision_speed
  check (jsonb_path_exists(config, '$.urgency.decision_speed ? (@ == "instant" || @ == "fast-track" || @ == "deliberative")'));

--  Complexity.tools_involved ∈ {single, multi-stack}
alter table parameter_sets
  add constraint ck_tools_involved
  check (jsonb_path_exists(config, '$.complexity.tools_involved ? (@ == "single" || @ == "multi-stack")'));

--  Output.validation: dacă include „score≥80 required”, conține și „file_format" in ("pdf","json","zip")
create or replace function f_ck_output_policy() returns trigger as $$
begin
  if (new.config #>> '{output,validation,0}') is not null
     and position('score≥80 required' in (new.config #>> '{output,validation}' )) > 0
     and (new.config #>> '{output,file_format}') !~ '^(pdf|json|zip)$' then
       raise exception 'score≥80 required este permis doar pentru pdf/json/zip';
  end if;
  return new;
end;
$$ language plpgsql;

drop trigger if exists trg_ck_output_policy on parameter_sets;
create trigger trg_ck_output_policy
before insert or update on parameter_sets
for each row execute function f_ck_output_policy();

3) Evaluator AI — scoruri calibrate by sub-parametri
a) Praguri dinamice (concept)

urgency=crisis → AmbiguityMax=0–10, ClarityMin=90, ExecutionMin=90; forțează secțiunea „Escalation Path”.

complexity.failure_cost=high → adaugă cerințe DoD: „Risks & Mitigations”, „Rollback Plan”.

application.delivery_mode=automated → cere „Triggers/Inputs/Outputs/Error Handling” (altfel FAIL DoR).

output.validation conține „score≥80 required” → blochează export pdf/json/zip sub 80.

b) Matrice de evaluare (pseudo-config)
{
  "thresholds": {
    "base": { "clarity": 80, "execution": 80, "ambiguity_max": 20, "business_fit": 75 },
    "urgency": {
      "crisis":   { "clarity": 90, "execution": 90, "ambiguity_max": 10, "business_fit": 80 },
      "sprint":   { "clarity": 85, "execution": 85, "ambiguity_max": 15 }
    },
    "complexity.failure_cost": {
      "high": { "clarity": 88, "execution": 88, "ambiguity_max": 12, "requires": ["risks", "rollback"] }
    },
    "application.delivery_mode": {
      "automated": { "requires": ["triggers","io","error_handling","owners","slo"] }
    }
  }
}

c) Verdict logic (pseudo-cod)
const pass = (scores, req) =>
  scores.clarity >= req.clarity &&
  scores.execution >= req.execution &&
  scores.ambiguity <= req.ambiguity_max &&
  scores.business_fit >= (req.business_fit ?? 75);

if (!hasRequiredSections(bundle, req.requires)) return FAIL_DOR;
if (!pass(scores, req)) return (composite >= 80 ? PARTIAL_PASS : FAIL_DOD);
if (output.validation?.includes("score≥80 required") && composite < 80) return FAIL_EXPORT;
return PASS;

4) Conectarea pieselor (pipeline, practic)

DB: parameter_sets ține 7D + sub-parametri în config (JSONB) cu câteva verificări SQL.

UI: Zod + RHF pentru formuri; gating de plan pe export; preseturi „mode”.

Evaluator: citește config și ridică/relaxează praguri; verifică secțiuni obligatorii; blochează exportul când regulile cer (DoR/DoD/Export gates).

Telemetry: loghează export_blocked_reason (ex: score_below_threshold, missing_required_section, plan_not_allowed) pentru funnel & upsell.
